## 1. 売り手側（こちらの営業・業務目標）

※これは **こちらの目標**。顧客が得るものではない。

### ゴール（12ヶ月イメージ）

社労士事務所向けに「残業チェック基幹＋必要ならAI下書き拡張」を、  
**スポット → 導入 → 保守** で継続課金できている状態。

数値目安:
- 有料顧客 5〜10事務所
- MRR 15〜30万円（保守中心）
- 初期導入の累計売上 150〜400万円
- 法令判断はせず、実装・改修・運用に閉じる

### 価格帯（階段）

| 段階 | 商材 | 価格目安 |
|------|------|----------|
| 0 | 無料診断ツール | 0円 |
| 1 | スポット試作（顧問先1社） | 8〜15万円 |
| 2 | 事務所導入 | 25〜40万円 |
| 3 | 月額保守 | 2〜3万円/月 |
| 4 | AI拡張（任意） | 初期+5〜15万、または月+0.5〜1万 |

### 売り手側シナリオ

無料リード → ヒアリング → スポット実証 → 導入 → 保守 →（任意）AI拡張

## 2. 顧客側（契約した社労士に何が残るか）

※こちらが本題。**先生の事務所に残る成果物・状態**の観点。

### 一文

> スタートは「Excelと感覚で残業を見ている状態」。  
> ゴールは「先生のルールで動くチェック資産が事務所に残り、顧問先対応が速く・再現可能になっている状態」。

### スタート（契約前〜初回）

先生の現状（よくある姿）:
- 顧問先の残業を Excel / 目視 / 経験で確認
- 法改正のたびに数式や手順が揺れる
- 「怪しい」は分かっても、根拠付きで即説明しづらい
- 職員に属人化し、先生本人がボトルネック

この時点で先生の手元にあるもの:
- 既存の勤怠CSVやExcel
- 自分の判断基準（頭の中／メモ）
- （無料ツール利用時）OvertimeChecker の一次スクリーニング結果

### 中間（スポット 1社で試作したあと）

先生の事務所に残るもの:
1. **動くチェック結果**（当該顧問先1社分）
   - 単月100h / 45h超過回数 / 複数月平均80h / 年720h などの判定一覧
2. **再現手順**
   - CSVを入れる → 結果が出る、という型
3. **顧問先への説明たたき台**（任意・推奨）
   - 違反論点を文章化した下書き（最終文言は先生が確定）
4. **「自事務所でも使える」実感**
   - 次に本導入するかどうか判断できる材料

まだ残らないもの:
- 全顧問先横断の運用基盤
- 法改正の継続追随（保守契約前）
- 法令保証そのもの（判断主体は常に先生）

### ゴール（導入＋保守まで進んだ状態）

先生の事務所に残るもの（資産）:

| 残るもの | 内容 |
|----------|------|
| **チェックツール本体** | 先生の運用に合わせた残業上限チェック（名入れ・項目合わせ後） |
| **事務所の手順** | 誰が・いつ・どのCSVで回すかの運用手順 |
| **再現可能な根拠** | 「なぜ引っ掛かったか」を画面/一覧で示せる状態 |
| **説明材料** | 顧問先への指摘・改善提案の型（テンプレ／過去出力） |
| **更新の受け皿** | 保守契約下で、法改正や軽い仕様変更を依頼できる関係 |

先生側のメリット（成果として感じる変化）:
- 確認時間が減る（Excel地獄の削減）
- 指摘の dedupe／抜け漏れが減る
- 職員でも一次チェックできる（属人化の緩和）
- 顧問先への説明が速くなる（信頼・単価交渉の材料になりうる）
- 「先生の考え方」がツールに残り、事務所の資産になる

### 顧客シナリオ（スタート → ゴール）

```text
【スタート】
Excel / 経験で残業確認。根拠提示に時間がかかる。

    ↓ 無料ツールで一次体験

【気づき】
「機械的に見られる部分がある」と分かる。
ただし無料版はお試し（人数制限など）。

    ↓ スポット（顧問先1社）

【試作成果が残る】
・1社分の違反一覧
・回し方の型
・指摘文のたたき台（任意）
→ 「本導入する価値があるか」を先生自身が判断できる

    ↓ 事務所導入

【資産が残る】
・自事務所向けに調整されたツール
・運用手順
・複数顧問先で使える状態

    ↓ 保守

【ゴール状態】
・毎月/定期で回る業務になっている
・法改正時も依頼先がある
・判断は先生、実行補助はツール、という役割分担が定着
```

### 具体シナリオ（36協定業務に沿った版）

顧客の一年と、ツールがどこで必要になるかの物語。

#### ① 年次：36協定の締結・更新（入口）

- 顧問先ごとに36協定を見直し／更新する（多くの事務所で年1回の山場）。
- 協定上の枠（例: 月45時間、特別条項、年720時間 など）と、**実際の残業実績がズレないか**が論点になる。
- この時点の先生の仕事は「協定を結ぶこと」だけでなく、**結んだあと守れているかを見続けること**。

ここで起きる痛み:
- 協定は結んだが、月次の実績チェックが Excel 属人
- 「特別条項を何回使ったか」を後から数えるのが辛い
- 顧問先に「今のペースだと危ない」を早めに言えない

#### ② だからこの種のツールが必要になる

36協定を「紙（届出）で終わらせず、運用で守る」ために必要になるもの:

| 必要なこと | ツールが出すもの |
|------------|------------------|
| 実績が上限に近づいていないか見る | 単月100h、複数月平均80h 等の一次判定 |
| 特別条項の使い過ぎを見る | 45h超過回数などの一覧 |
| 年通しで見る | 年720h などの累計チェック |
| 顧問先に説明する | 違反／予兆一覧 → 指摘文のたたき台 |

つまりツールの位置づけは:

> **36協定の“締結”を代替するものではない。**  
> **締結した協定を、月次の実績で壊れないように監視する装置。**

#### ③ 月次：運用フェーズ（ここが本丸）

導入後の毎月（または給与前）:

1. 顧問先（または勤怠）から CSV を受領  
2. ツールに投入  
3. 引っ掛かった人・予兆を確認（先生または職員）  
4. 必要なら顧問先へ早期指摘・働き方の改善提案  
5. 問題なければ次月へ

この繰り返しが回ると、顧客側のゴール状態になる:

- 「協定更新のときだけ焦る」から「毎月淡々と守る」へ
- チェックが再現可能（誰がやっても同じ基準）
- 説明の材料が事務所に蓄積される

#### ④ 年1回前後：法改正・通達・運用見直しのとき

法令や解釈、実務運用が変わった／変わりそうなタイミング（年1回とは限らないが、事務所にとっては年次イベントになりやすい）:

例:
- 上限規制の周知強化、監督実務の変化
- 顧問先の働き方変更（変形労働・シフト変更）に合わせた見方の追加
- 事務所としてのチェック項目の追加（先生が「これも見たい」と言い出す）

このとき顧客に残るメリット（保守がある場合）:

```text
【法改正・ルール変更が発生】
        ↓
先生が「新しい見方」を決める（判断は先生）
        ↓
こちらがツールの判定ロジック／表示を改修
        ↓
翌月から同じCSV手順のまま、新しい基準で回る
```

Excelだけだと起きがちなこと:
- 数式を直した人しか分からない
- 顧問先ごとにシートが分岐して壊れる
- 「いつの版のルールで見ているか」が不明になる

ツール＋保守があると残るもの:
- **更新された判定資産**（事務所の標準）
- **同じ運用手順**（現場が混乱しにくい）
- **変更履歴の受け皿**（何をいつ変えたか依頼ベースで残せる）

#### ⑤ ゴール状態（顧客視点の到達点）

1. **36協定サイクル**（年次更新）と **実績監視サイクル**（月次）がつながっている  
2. 監視はツール、最終判断・顧問先助言は先生、と役割が分かれている  
3. 法改正時は「Excelを全書き換え」ではなく「基準を一箇所更新」で済む  
4. 事務所に、再現可能なチェック資産と説明の型が残っている  

#### 時系列の一枚絵

```text
[年次] 36協定の締結・更新
   │
   │  「結んだあと、どう守る？」
   ▼
[必要化] 実績モニタリング手段が要る
   │
   │  無料体験 → 1社スポット → 事務所導入
   ▼
[月次] CSV投入 → 判定 → 早期指摘 → 改善
   │                    ▲
   │                    │ 同じ手順で回す
   ▼                    │
[随時/年次] 法改正・ルール追加 ──► ロジック更新（保守）
   │
   ▼
[ゴール] 協定運用が、属人Excelから事務所の標準業務になる
```

---

約束してよい:
- 残業上限の一次チェックを高速化・再現可能にすること
- 先生の指定ルールをツールに落として事務所に残すこと
- 運用手順と改修の受け皿を用意すること

約束しない（判断は先生）:
- 「これで法令適合が保証される」
- 労基署対応・就業規則の適法レビューの代行
- AIが最終判断すること

## 4. 「36協定だけでは1年持つのか」と隣接領域

### 結論

- **月次運用としては1年持つ。** 協定更新は年1回でも、実績監視は毎月ある。
- **商材の物語・拡張・法改正ネタとしては薄い。** 上限規制の大きな改正は頻繁ではないので、「毎年ドカンとルールが変わる」前提で保守を売るのは弱い。
- だから入口は36協定監視でよいが、**隣のExcel地獄に伸ばせると持ちがよくなる。**

### 36協定監視だけで回るもの / 回らないもの

| 持つ | 弱い |
|------|------|
| 毎月のCSVチェック（本丸の利用頻度） | 「今年の法改正対応」だけを価値の中心にする話 |
| 顧問先への早期指摘の反復 | 協定書作成そのもの（文書作成が主業務の先生には刺さりにくい） |
| 職員への一次チェック委譲 | すでに勤怠クラウドで上限アラートが出ている事務所 |

### 社労士が Excel / ドキュメントで詰まりやすい隣接領域（拡張候補）

優先度は「今のツールからの距離」と「痛みの強さ」の目安。

| 領域 | よくある作業 | ツール化との相性 | メモ |
|------|--------------|------------------|------|
| **① 残業上限（36系）** | 100h / 45h回数 / 平均80h / 年720h | ◎（現行） | 入口商材 |
| **② 割増賃金の検算** | 法定内/外、60h超、休日深夜の単価確認 | ○ | 給与ソフトと競合しやすい。スポット検算向き |
| **③ 年休5日義務** | 付与・消化・時季指定の管理表 | ◎ | 年次＋月次。Excel管理が多い |
| **④ 変形労働の枠** | カレンダー／総枠／週・月の過不足 | ○〜◎ | ロジックが重い。差別化しやすい |
| **⑤ 算定基礎・年度更新** | 年1回の集計・届出前チェック | △〜○ | 季節商材。単発支援向き |
| **⑥ 就業規則と実態の差分** | 規定と勤怠・運用の突合（チェックリスト） | △ | ドキュメント＋ヒアリング寄り。AI下書きと相性あり |
| **⑦ 育児介護・休業給付まわり** | 期間・給付・手続きスケジュール | △ | 案件型。常時ツール化しにくい |
| **⑧ 助成金要件管理** | 要件チェックリスト、証拠書類 | △ | 文書多い。生成AI＋チェックリスト向き |
| **⑨ 同一労働同一賃金の整理** | 待遇差の説明材料、比較表 | △ | コンサル色が強い |
| **⑩ 労働時間の適正把握全般** | 出勤簿・自己申告の乖離確認 | ○ | ①の前後工程として提案しやすい |

### 現実的な伸ばし方（顧客シナリオの拡張版）

```text
Year 0-1  入口: ① 36協定＝残業上限の月次監視
          └─ これだけで「毎月使う理由」は作れる

Year 1    横展開A: ③ 年休5日（同じ勤怠データを別観点で見る）
          横展開B: ⑩ 把握漏れ（打刻と自己申告の差など）

Year 1-2  深掘り: ④ 変形労働など、顧問先タイプ別ロジック
          単発: ⑤ 算定・年度更新の前チェック

並行      ② は「給与計算を代替」ではなく「怪しい明細の検算」に限定すると衝突しにくい
```

### 営業トーク上の使い方

- **初年度の保守説明:** 「法改正があるから」より **「毎月の監視を止めない／項目追加に応える」** を主にする。
- **更新提案:** 法改正待ちにせず、年休・変形・把握 など **隣のExcel表を1枚潰す** 提案にする。
- **約束の境界は維持:** どの領域でも、最終の法令判断・顧問先助言は先生。

## 5. 作り方（ヒアリング駆動か）と届け方（Excel / Sheets / Python）

### 結論（先に）

- **10項目の多くは、ヒアリングしながら型を作っていける業務**（先生のルールを聞き、表または判定に落とす）。
- **社労士側の日常UIとしては、Googleスプレッドシート（またはExcel）の方がハードルが低い。**
- **Mac/Windows の Python 配布は、技術的には正しいが、士業事務所のスキル・運用文化には重い**可能性が高い。
- 現実解: **ロジックはこちらが持つ／先生は使い慣れた表またはブラウザで触る。**

### ヒアリングしながら作れるか

| 領域 | ヒアリング駆動 | 主な成果物の置き場 |
|------|----------------|--------------------|
| ① 残業上限 | ◎（判定基準はほぼ標準、CSV列合わせが主） | Sheetsでも可。件数・履歴が多いとアプリ/Webが楽 |
| ② 割増検算 | ○（単価・手当の扱いを聞く） | **Sheets/Excelが本命** |
| ③ 年休5日 | ◎（付与ルール・管理単位を聞く） | **Sheetsが本命** |
| ④ 変形労働 | ○〜△（カレンダー設計が重い） | 最初Sheets試作 → 複雑化したらアプリ/Web |
| ⑤ 算定・年度更新 | ○（年1の手順を聞く） | **Sheets/Excelで十分なことが多い** |
| ⑥ 就業規則×実態 | ◎（チェックリスト化） | Docs + Sheets。AI下書き向き |
| ⑦ 育児介護 | ○（案件ごとに違う） | チェックリスト／スケジュール表 |
| ⑧ 助成金 | ◎（要件を表にする） | Sheets + フォルダ設計 |
| ⑨ 同一労働同一賃金 | ○（説明材料の型） | Sheets比較表 + 文書 |
| ⑩ 適正把握 | ○（何と何を突合するか聞く） | Sheets。①とデータを共有しやすい |

要するに:

> **「法令をゼロから発明する」のではなく、「先生の既存Excel作業を、壊れにくい型にする」**ので、ほとんどがヒアリング駆動で作れる。

### Excel必須か？ Sheetsで収まるか？

| 置き場 | 向くこと | 向かないこと |
|--------|----------|--------------|
| **Googleスプレッドシート** | 共同編集、事務所内共有、改修が速い、学習コスト低 | 超大規模・複雑な状態機械、オフライン必須の事務所 |
| **Excel** | 先生が既にそれしか使っていない、マクロ資産がある | 版管理が壊れやすい、远程更新しづらい |
| **Pythonアプリ（現行）** | 複雑判定、ローカル完結、配布の「製品感」 | 起動・パス・セキュリティ警告・職員教育 |
| **ブラウザWeb** | インストール不要、更新が一発 | 開発・ホスティング・個人情報設計が要る |

目安:
- **③⑤⑥⑦⑧⑨⑩、②の多く → Sheets枠で十分なことが多い**
- **①④（と②の高度版）→ Sheetsで試作し、痛くなったらWeb/アプリへ**

### Python配布ハードルについての判断

想定される摩擦:
- 「`.bat` / `.app` を開いてよいのか」不安（Windows SmartScreen、macOSゲートキーパー）
- 職員がローカルサーバ（`localhost`）の概念を持たない
- 顧問先データをローカルに置く運用ルールが事務所ごとに違う
- トラブル時に「画面が黒い」「起動しない」で止まり、非士業のあなたが遠隔サポートしづらい

だから:

> **無料の体験・デモ用に Python 配布はあり。**  
> **本導入の本命UIを最初から Python 前提にしない方が安全。**

推奨レイヤ分け:

```text
[先生・職員が触る]
  Googleスプレッドシート / またはブラウザ
        ↑ データ入出力
[こちらが守るロジック]
  判定ルール・バージョン・改修（Scripts / 裏方Python / Pages Functions 等）
```

スポット提案の言い方例:
- 「先生は今のCSVをシートに貼るだけ」
- 「判定と更新はこちらが持つ」
- 「将来、件数や複雑さが増えたら画面をWebに上げる」

### 売り手制約（Mac前提）と、Sheets/GAS再構築の判断

前提:
- こちら（売り手）は **Mac** のため、顧客向けに Excel マクロ中心で作るのは辛い
- 共同編集・遠隔改修・自分の作業環境の一致を考えると **Googleスプレッドシート + GAS が最有力**
- 現行の Mac/Windows Python 配布は「動く製品」ではあるが、**顧客スキルと売り手の保守コストの両方で重い**

#### Sheets/GAS で収まりやすいか（現行①残業上限）

| 要素 | Sheets/GAS | メモ |
|------|------------|------|
| CSV貼り付け → 判定一覧 | ◎ | 本丸。Import or 貼付タブで十分 |
| 100h / 45h回数 / 平均80h / 年720h | ◎ | GAS or 数式で再現可能 |
| 10名お試し制限 | ◎ | シート側で行数制限すればよい |
| ローカル完結・オフライン | △ | ネット必須。機密ポリシー確認が必要 |
| 「アプリ感」のある配布 | △ | 代わりに「事務所専用シートを共有」になる |
| Windows/Mac差のサポート | ◎ | ブラウザならほぼ消える |

#### 再構築するならの現実的な形

```text
[顧客]
  事務所用スプレッドシート（コピー配布 or 共有）
  └ 勤怠貼付タブ → 結果タブ →（任意）指摘文下書きタブ

[売り手]
  マスター雛形を改修 → 顧客シートへ反映（コピー更新 or ライブラリ化）
  GASで判定・整形・簡単なUI（カスタムメニュー）
```

価格・提案との相性:
- スポット（8〜15万）: **顧問先1社分のシートを一緒に作る** が一番わかりやすい
- 導入（25〜40万）: 事務所用テンプレ＋運用手順＋名入れ
- 保守（2〜3万/月）: 判定ロジック更新、列追加、問い合わせ

#### いつ作り直すか（判断基準）

**Sheets/GASへ寄せる（推奨寄り）** if:
- 本導入の主戦場を「先生が毎週/毎月自分で回す」にする
- サポートコストを下げたい（bat/.app起動切れを減らす）
- 隣の③年休などを同じ土台で横展開したい

**Python配布を残す** if:
- ネット不可・Google禁止の事務所が主顧客
- ローカル完結が契約条件
- 無料LPの「ダウンロードして動く」デモ体験を強く残したい

併存案（現実的）:
1. **LP無料:** 今の ZIP（体験・営業フック）は残してもよい  
2. **有料本線:** Sheets/GAS に寄せる  
3. 機密で Google 不可なら、その案件だけ個別対応（例外）

#### 注意点（Sheets/GASにするなら最初に決める）

- 顧問先の個人情報を Google に載せてよいか（事務所の方針確認）
- オーナーシップ（顧客オーナー / こちらオーナー）と退職・解約時の扱い
- 数式だけに寄せすぎない（壊れる）。**判定の核心は GAS に閉じる**方が保守しやすい
- 「脱Excel」LP表現との整合（Sheetsも表計算、だが共同編集と更新性で差別化できる）

#### Google無料枠で全部いけるか


結論:
- **試作・スポット数件・手動実行中心なら、個人Googleアカウントの無料枠でほぼ足りる**
- **「全顧客・全自動化・長期の本番基盤を無料だけで」は厳しい／おすすめしない**

無料（個人Gmail等）で足りやすいもの:
- スプレッドシート本体、共有、カスタムメニューからのGAS実行
- CSV貼付 → 判定 → 結果タブ（実行時間数分以内の処理）
- 顧客ごとにファイルをコピーして渡す運用

無料で詰まりやすい／注意が要るもの:

| 項目 | 目安 | 影響 |
|------|------|------|
| Drive容量 | 合計15GB（Gmail等と共有） | CSV・旧版シートが増えると逼迫 |
| GAS実行時間 | 1回あたり約6分 | 大人数・複数月の一括で苦しくなることあり |
| トリガー／自動実行の枠 | 消費者枠は Workspaceより狭い | 「毎朝自動チェック」を多顧客で回すと危険 |
| メール送信枠 | 1日あたり上限あり | 結果通知を自動メールにするとすぐ逼迫 |
| ビジネス体裁・契約 | 個人アカウント前提 | 顧問先PIIを預ける説明が弱い。Workspaceの方が無難 |

実務的な分け方:
1. **今（検証〜最初の数件）:** 無料Googleで十分。コストをかけてよいのは学習と型作り
2. **有料顧客が継続し始めたら:** 売り手側は **Google Workspace（有料）** を検討（管理・共有・信頼性）
3. **顧客事務所側:** 可能なら顧客のWorkspace／Googleアカウントをオーナーにし、こちらは編集者（PIIの所在を顧客側に残す）

「全部無料で永続」より、

> **無料で型を証明 → 顧客が増えたら Workspace に上げる**

が安全。

## 7. 顧客アカウントへの展開: 手順化 /（半）自動化は可能か

### 結論

**可能。ただし「完全ワンクリック無人デプロイ」は現実的でない。**  
現実解は **マスター雛形のコピー＋（半）自動のGAS同期**。

10項目すべてを毎回ゼロから顧客テナントに組むのは避け、  
**コピー可能なテンプレ（シート＋GAS）を資産化**するのが前提。

### 難易度イメージ

| 方式 | 自動化度 | 向く場面 | 注意 |
|------|----------|----------|------|
| **A. 「コピーを作成」リンク** | ○（手順化しやすい） | スポット〜初期の本命 | 顧客がコピーするだけ。GASも一緒にコピーされる（再承認は顧客側） |
| **B. こちらが顧客ドライブに雛形を作成** | ○〜◎ | 伴走導入 | 顧客から編集者権限をもらい、テンプレを配置。clasp/手動どちらも可 |
| **C. clasp で顧客プロジェクトへ push** | ◎（改修の半自動） | 保守・バージョン更新 | 顧客の scriptId を台帳管理。初回は認証・デプロイ設定が必要 |
| **D. Workspace Marketplace / アドオン** | ◎◎ | 将来の横展開 | 審査・掲載コスト。今は早すぎる |
| **E. ドメイン全体委任＋API一括** | ◎◎ | Workspace大規模 | 個人Gmail顧客には不可。過剰 |

### 推奨オペ（いま〜近いうち）

```text
1. マスター（自社デモGoogle）で①〜⑩の完成度を上げる
2. 顧客ごとに「コピー用マスター」URLを渡す  ← A
   または 共有フォルダにこちらが配置       ← B
3. 初回: 顧客がGASの権限許可（画面から1回）
4. 以降の改修: clasp で該当 scriptId に push ← C（半自動）
5. WebアプリURLが必要なら、顧客テナントでデプロイ（手順書化）
```

### 10項目をどう扱うか

- **全部を最初から顧客に入れない。** ヒアリングで選んだ1〜2個だけ展開。
- ①（残業上限）は Web UI 込みの完成テンプレとしてコピー展開向き。
- ②〜⑩は「シート雛形＋薄いGAS」を同様にコピー展開。必要になったら厚くする。

### 半自動化で用意すると効くもの（チェックリスト）

- [ ] マスターシート（領域ごと）の「コピー用」固定URL
- [ ] 顧客台帳: 事務所名 / スプレッドシートID / scriptId / 導入領域 / 最終push日
- [ ] `clasp push` 用の領域別プロジェクト（または monorepo の folder）
- [ ] 顧客向け1枚手順: コピー → 権限許可 →（任意）Webアプリデプロイ
- [ ] 改修時の「全顧客へ順次 push」スクリプト（IDリストを読む）

### いまやらない方がいいこと

- 顧客のGoogleパスワードを預る
- 全顧客を自社の単一Webアプリに集約する（PII方針と矛盾）
- Marketplace化を初手にする

### 一文

> **デプロイの本体は「顧客がコピー（またはこちらが配置）」で手順化でき、  
> その後の改修は clasp による半自動 push まで持っていける。**

### GitHubアカウントを顧客手順に入れるか

結論: **基本は入れない。入れるなら「希望する事務所だけ・任意」に留める。**

| 入れた場合にできること | 実態 |
|------------------------|------|
| 顧客リポジトリにGASソースを置く | バージョン履歴が顧客側に残る |
| PRで改修レビュー | エンジニア寄り事務所には刺さる |
| Actionsで clasp push | 半自動更新の本丸になりうる |

| 入れない方がよい理由 | 実態 |
|----------------------|------|
| 士業にGitHubは馴染みが薄い | アカウント作成だけで離脱しやすい |
| 本件の本丸はSheets運用 | 顧客が触るのは表と画面で足りる |
| 改修の半自動化はこちら側で足りる | **こちらのGitHub＋顧客scriptId台帳＋clasp** で回せる |

推奨の役割分担:

```text
[こちら]
  GitHub でソース管理（samples / gas）
  clasp で顧客の scriptId へ push

[顧客]
  Google（Sheets / 権限許可）だけ用意
  ※GitHubは不要（原則）
```

もし手順に書くなら、こう分ける:

1. **必須:** Googleアカウント（できればWorkspace）
2. **任意・上級:** GitHubアカウント  
   - 「ソースも自事務所で持ちたい／将来内製したい」事務所向け  
   - こちらが Private repo を共有 or 顧客repoへPRする形

一文:

> **顧客に作ってもらうのはGoogle。GitHubは売りの側の道具。**

### こちらから顧客Googleへデプロイする方法はあるか

**ある。** ただし「顧客のパスワードなしで、勝手に全権限」は不可。  
**顧客が一度だけ権限を渡す**前提なら、配置はこちら主導にできる。

#### 現実的な手段（強い順）

**1. 共有フォルダ方式（いちばん現実的）**

```text
顧客: Driveにフォルダを作り、こちらを「編集者」で招待
こちら: マスターをそのフォルダへコピー／新規作成
       GASを push（clasp）
       必要なら所有権を顧客へ移管
```

- 顧客作業: フォルダ作成＋共有（数分）
- デプロイ作業の本体はこちら
- 個人Gmailでも可

**2. 既存シートへの紐づけ**

```text
顧客: 空のスプレッドシートを作り、こちらを編集者に
こちら: その spreadsheetId を parent にして GAS を作成・push
```

- Web UI が必要なら、顧客テナント上でデプロイ設定（または手順書で1回やってもらう）

**3. ワンショット認可インストーラ（半自動・将来）**

```text
顧客が専用リンクを開き「許可」を1回押す
→ こちらが OAuth トークンで Sheets/Apps Script API 経由デプロイ
```

- いちばん「こちらからデプロイ」感が強い
- 実装・審査・トークン管理のコストあり（今は後回しでよい）

**4. Workspaceのドメイン委任**

- 顧客が Workspace で、かつ同じ組織の管理者権限が必要
- 一般の社労士事務所（別組織・個人Gmail）には使えない

#### できない／避けること

- 顧客パスワードでログインして代行
- 見知らぬ他人のGoogleへ無許可デプロイ
- サービスアカウントだけで個人Gmailへ勝手に書き込み

#### 推奨オペ（「こちらデプロイ」寄り）

```text
1. 契約後、顧客に「共有フォルダ作成＆編集者招待」だけ依頼（手順1枚）
2. こちらがテンプレ配置＋GAS push＋動作確認
3. 所有権を顧客へ移管（資産は顧客のもの）
4. 以降の改修は scriptId 台帳を見て clasp push
```

一文:

> **顧客にやってもらうのは「箱（フォルダ／権限）」だけ。中身のデプロイはこちらからできる。**

### Cursor から提供する

可能。流れは次のとおり。

1. 顧客がシート（またはフォルダ）を編集者共有  
2. Cursor にスプレッドシートIDを渡す  
3. エージェントが `samples/uplink-sharoushi/scripts/deploy-01-to-client.sh` を実行  
4. `clients/<slug>.json` に記録 → 以降は `push-01-client.sh` で改修

詳細手順: `samples/uplink-sharoushi/DEPLOY.md`

### 契約者ごとのソース履歴は、今のGitHubリポジトリだけで足りるか

**基本は足りる。** ただし「何を履歴に残すか」を分ける。

| 残すもの | 今のリポジトリで | 置き場の例 |
|----------|------------------|------------|
| 共通テンプレ（①〜⑩） | ◎ | `samples/uplink-sharoushi/01-...` 等 |
| 顧客へのデプロイ記録（ID・日時） | ◎ | `samples/uplink-sharoushi/clients/<slug>.json` |
| 顧客向けに**少しカスタムしたGAS** | ○ | `clients/<slug>/src/01-overtime-36/` を追加して管理 |
| 顧問先の実CSV・個人情報 | ✕（入れない） | 顧客Googleのみ |
| Apps Script 上の実行バージョン | △（別系統） | 顧客のスクリプトの「バージョン履歴」＋こちらは git tag |

推奨ルール（1リポジトリ運用）:

```text
samples/uplink-sharoushi/
  01-overtime-36/          … 共通マスター（全員の起点）
  clients/
    yamada-office.json     … 台帳（spreadsheetId / scriptId）
    yamada-office/         … 必要なら顧客差分ソースのみ
      src/01-overtime-36/
```

- **差分がなければ** `clients/<slug>.json` だけでよい（ソースはマスターを push）
- **列名・判定・名入れで分岐したら** そのとき初めて `clients/<slug>/src/` にコピーして git 管理
- コミットメッセージ例: `deploy(yamada-office): ① overtime UI v5`

やらなくてよいこと:
- 契約者ごとに GitHub リポジトリを増やす（運用が壊れる）
- 顧客データを git に入れる

一文:

> **履歴の本体はこのGitHub1本でよい。顧客台帳＋（必要なときだけ）顧客差分フォルダ。実データはGoogle側。**
